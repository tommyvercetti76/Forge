# Madhubani Tee Catalog — Workflow Reference

The catalog is built **one animal at a time**. A "set" = 1 animal × 4
poses = 4 designs reviewed together. We do not batch. We do not
multi-seed-and-pick.

This document is the operational reference. For the *why* behind these
choices, read `CATALOG_PLAN.md` §6.

---

## The loop

```
┌──── Pick next animal from brand/madhubani/animals.json ────┐
│                                                            │
▼                                                            │
1. RENDER set       — all 4 poses, 1 seed each (~5-7 min GPU)
   Output: generated/madhubani_animals/attempts/{slug}/v{N}/
   Receipts: render-manifest.json + generated/madhubani_animals/workflow-events.jsonl
   │
   ▼
2. REVIEW           — open all 4 PNGs side by side, score against RUBRIC.md
   │
   ▼
3. IDENTIFY GAPS    — for each pose, document: which checks failed, what
                      to change (different seed? per-pose subject tweak?
                      engine rule needed?)
   │
   ▼
4. RETRY (once)     — re-render gapped poses with adjustments:
                       (a) different seed, OR
                       (b) per-pose subject string tweak, OR
                       (c) both
                      Maximum 1 retry per pose. Output goes to v{N+1}/.
   │
   ▼
5. REVIEW v2        — same rubric on the retried renders
   │
   ▼
6. DECISION         — for each of the 4 poses, one of:
                       ✓ MASTERED → promote to mastered/{slug}/{pose}/
                       ⚑ FLAGGED  → keep best attempt in flagged/{slug}/{pose}/
                                   with FLAG_NOTES.md
                       ✗ REJECTED → leave in attempts/, move on
   │
   ▼
7. ARTIST CARD      — generate mastered/{slug}/ARTIST_CARD.md
                      (auto-generated by bin/forge_madhubani.py from
                       directive sidecars + masters.json + animals.json)
   │
   ▼
8. UPDATE INDEX     — generated/madhubani_animals/INDEX.md gets a
                      mastered++ or flagged++ tick
   │
   ▼
9. NEXT ANIMAL      ─────────────────────────────────────┘
```

---

## Why retry-once-then-flag (not retry-until-perfect)

Open-ended iteration is how 4 designs becomes a week of babysitting.
The retry-once gate forces a binary call: *the prompt + engine can
produce this, or it can't yet.* If it can't on retry, we **flag**
(keep the best attempt aside with notes) and move on.

Flagged animals queue for a second pass after we've mastered the
engine further — usually after a LoRA training cycle or a prompt
revision triggered by patterns in the flag pile.

This is also how you avoid burning emotional energy on the hard cases
while easier wins are sitting idle.

## Directory structure for one animal

```
generated/madhubani_animals/
├── attempts/{slug}/                ← every attempt parked here
│   ├── v1/
│   │   ├── 01_{slug}_standing-alert.png
│   │   ├── 01_{slug}_standing-alert.png.directive.json
│   │   ├── 01_{slug}_standing-alert.qc.json
│   │   ├── 01_{slug}_standing-alert.transparent.png
│   │   ├── 02_{slug}_seated-rest.png
│   │   ├── ... (4 poses × render/directive/QC/transparent files)
│   │   ├── render-manifest.json      ← machine-readable render set receipt
│   │   └── REVIEW_NOTES.md         ← Rohan's notes from step 2
│   └── v2/                         ← only if retry happened
│       └── (just the retried poses, not full set)
│
├── mastered/{slug}/                ← only mastered poses promoted here
│   ├── standing-alert/
│   │   ├── print-1280.png
│   │   ├── print-5120.png          ← optional upscaled production file
│   │   ├── transparent-1280.png
│   │   ├── transparent-5120.png    ← optional upscaled transparent file
│   │   ├── directive.json
│   │   └── auto-qc.json
│   ├── seated-rest/  ← if mastered
│   ├── ARTIST_CARD.md              ← set-level card (covers all mastered poses)
│   └── mockups/                    ← cream + black tee mockups (forthcoming)
│
└── flagged/{slug}/                 ← poses that couldn't be mastered
    ├── frontal-portrait/
    │   ├── best-attempt.png
    │   ├── directive.json
    │   ├── auto-qc.json
    │   └── FLAG_NOTES.md           ← what failed, what to try later
    └── ...
```

Every render/promote/flag/card action also appends one JSONL row to:

```text
generated/madhubani_animals/workflow-events.jsonl
```

Every successful render also writes a per-pose auto-QC receipt:

```text
attempts/{slug}/v{N}/{ordinal}_{slug}_{pose}.qc.json
```

That receipt covers the four machine-scorable rubric gates: palette floor,
clean corners, centered subject, and saturated body fill. `promote` refuses to
master a pose with failed auto-QC unless `--force` is used after human review.

## How to record outcomes

After step 5 (REVIEW v2), update the animal's set folder:

1. **For each MASTERED pose** — run `bin/forge_madhubani.py promote`.
   This copies the winning PNG, transparent sibling, directive, and auto-QC into
   `mastered/{slug}/{pose}/` and appends a workflow event. Upscale to 5120 is
   optional follow-up until the production upscale command is wired into
   `promote`.
2. **For each FLAGGED pose** — copy the best attempt into
   `flagged/{slug}/{pose}/`, write `FLAG_NOTES.md` with: which checks
   failed, hypotheses for why, what to try in the next pass.
3. **For each REJECTED pose** — leave in `attempts/`, document in
   `REVIEW_NOTES.md` that we're moving on.
4. **Update** `generated/madhubani_animals/INDEX.md` with the new state. The
   event log is automatic; the human-readable index still needs review.

---

## When to break the loop (escalation triggers)

- **3 animals in a row hit ≥2 flags each** → don't start the next
  animal; queue a "prompt revision pass" instead. Look at the flag
  notes, identify the common pattern, update the engine or the
  prompt grammar, then return to the loop.
- **Same pose fails across multiple species** → the pose template
  itself may need revision in `poses.json`.
- **Color floor failing on >50% of attempts** → palette anchor for
  that body-type may be wrong; revisit `animals.json` `body_fill_color`.
- **Reached 12 mastered animals** → queue LoRA training (Catalog Plan §7).

## Quick command reference

```bash
# Render a full 4-pose set:
python bin/forge_madhubani.py render rhino --all-poses

# Render a full set with parallel Metal workers (runtime still caps by memory):
python bin/forge_madhubani.py render rhino --all-poses --jobs 2

# Render a single pose retry:
python bin/forge_madhubani.py render rhino frontal-portrait --retry

# Promote a mastered pose:
python bin/forge_madhubani.py promote rhino standing-alert --from-version v1

# Promote after human override of failed auto-QC:
python bin/forge_madhubani.py promote rhino standing-alert --from-version v1 --force

# Flag a pose:
python bin/forge_madhubani.py flag rhino frontal-portrait --from-version v1 --notes "face proportion off, try after engine revision"

# Generate artist card for a set:
python bin/forge_madhubani.py card rhino
```

For exact-stroke minimalist exploration outside the Madhubani catalog:

```bash
python bin/forge.py minimal-animal --animal "alert tiger in side profile" --max-lines 8
```
